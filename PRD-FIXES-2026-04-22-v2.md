# PRD — Fire TV Dashboard Reliability Fixes (v2)

**Date:** 2026-04-22
**Owner:** Platform / FireTV
**Target release:** next CI build after merge
**Scope:** bug-fix release — no new features, no architecture change.

---

## 1. Problem

The Fire TV dashboard APK installs and launches, but is not reliable for 24/7 wall-display use. Four defects cause the majority of visible failures:

1. **Forced reload loop every ~90 s.** The JS watchdog looks for an `AndroidDashboard` JavaScript bridge that is never bound via `addJavascriptInterface`. `lastJsPongMs` is therefore never updated from JS, and the watchdog fires `onMainFrameError` forever. Symptom: dashboard blanks and reloads roughly every 90–120 seconds.
2. **No recovery after WebView renderer crash.** `onRenderProcessGone` calls `webView.reload()` on an already-dead WebView. Symptom: black screen until Fire Stick is rebooted.
3. **No auto-resume after power-loss reboot** on Fire OS 7+. `BootReceiver` starts an Activity directly from a background `BroadcastReceiver`, which is blocked by OS background-activity-start rules. Symptom: after a power cut someone has to pick up the remote and relaunch.
4. **Gzip double-decompression** in the proxy interceptor. `HttpURLConnection` auto-decompresses gzip but the `Content-Encoding: gzip` header is forwarded unchanged; WebView then tries to gunzip plain bytes. Symptom: intermittent JS parse errors / blank page after any deploy that changes compressible assets.

Plus a set of smaller defects: over-aggressive `onTrimMemory` reloads on low-memory Fire Stick 4K Gen 1, wake-lock leaks on exit, no crash logging, `FLAG_SECURE` blocking benign screen mirroring, focus landing on overlay buttons at startup, `clearCache(true)` wiping the disk cache on every exit.

## 2. Goals

| # | Goal | Success metric |
|---|------|---------------|
| G1 | Zero forced reloads when the dashboard and network are healthy | No `JS watchdog TRIGGERED` log line for ≥ 6 h of uptime |
| G2 | Automatic recovery from renderer crash without user action | After `adb shell am crash com.hermes.firetv:sandboxed_process0`, dashboard is visible again within 10 s |
| G3 | Auto-resume on Fire OS 7 / 8 after power-loss | After `adb reboot`, the dashboard is visible within 60 s of the launcher appearing |
| G4 | No gzip-related blank pages | `curl -H "Accept-Encoding: gzip" https://dashboard.cashlabnyc.com/` returns gzip; dashboard renders OK in WebView |
| G5 | Survive 12 h overnight soak on Fire Stick 4K Gen 1 (1 GB RAM) | No reload loop, no blank screen, memory drift ≤ 20 % |
| G6 | Actionable crash diagnostics for remote devices | Uncaught exceptions persisted to `filesDir/crash-*.log` and POSTed on next launch |

## 3. Non-goals

- Kiosk lockdown (Device Owner provisioning). Out of scope — requires ADB provisioning per device.
- OTA auto-update flow. Tracked separately.
- Per-device auth token provisioning. Tracked separately — current APK-embedded token is unchanged.
- Rewriting `DashboardActivity` into smaller classes. Deferred.

## 4. Scope of changes

### 4.1 `DashboardActivity.kt`

- **Bind JS bridge:** call `webView.addJavascriptInterface(this, "AndroidDashboard")` in `setupViews()` before `loadDashboard()`.
- **Watchdog tuning:** raise `STALE_THRESHOLD_MS` from 90_000 to 180_000. Require **two consecutive** misses before declaring the page unresponsive. Remove the double-increment of `consecutiveErrors` in `checkJsHealth`.
- **Renderer crash recovery:** in `onRenderProcessGone`, detach the dead WebView from `rootView`, call `destroy()`, build a fresh WebView via a new `rebuildWebView()` helper, and reload.
- **Memory pressure:** only reload on `TRIM_MEMORY_COMPLETE` (80). Call `webView.freeMemory()` before `reload()`.
- **Exit cleanup:** stop `KeepAwakeService` before `finish()`.
- **Focus:** set `webView.isFocusableInTouchMode = true` and `webView.requestFocus()` after `loadDashboard()`. Make exit + reload buttons `isFocusable = false`.
- **Cache retention:** remove `clearCache(true)` from `onDestroy`.
- **`FLAG_SECURE` removal:** delete; a wall dashboard does not need anti-screenshot protection, and it causes benign mirroring problems on some TVs.
- **Proxy hardening:** add `content-encoding`, `content-length` to `HOP_BY_HOP`; force `Accept-Encoding: identity`; preserve WebView's UA by omitting the `User-Agent` override.
- **Rename typo:** `MAX_ERRORS_BEFORE_STopping` → `MAX_ERRORS_BEFORE_STOP`.

### 4.2 `BootReceiver.kt`

- Replace direct `startActivity` with a `startForegroundService(KeepAwakeService)` call, passing an extra `launchActivityOnStart=true`. The service brings the Activity up once it is itself foregrounded.

### 4.3 `KeepAwakeService.kt`

- Read the new intent extra; if `launchActivityOnStart`, call `startActivity(DashboardActivity, FLAG_ACTIVITY_NEW_TASK)` at the end of `onStartCommand`. This is permitted because the service is running foreground and the call happens during the foregrounding window.
- No wake-lock-logic changes (PARTIAL_WAKE_LOCK remains correct; `FLAG_KEEP_SCREEN_ON` on the Activity handles screen).

### 4.4 `FireTVApplication.kt`

- Install a `Thread.setDefaultUncaughtExceptionHandler` that writes `filesDir/crash-<ts>.log` and then delegates to the previous handler.
- On next `onCreate`, scan `filesDir` for `crash-*.log` and POST each one to `https://dashboard.cashlabnyc.com/api/crash` with the `X-App-Auth` header and `BuildConfig.VERSION_CODE`, deleting files on success. Offloaded to a background thread; failures are swallowed silently (the log persists to the next attempt).

### 4.5 `AndroidManifest.xml`

- Add `<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />` (required on API 33+; harmless below).

### 4.6 Dead-code / hygiene

- Delete `res/layout/activity_main.xml` (never inflated).
- Remove the stale `-keep class ch.acra.**` block from `proguard-rules.pro`.

## 5. Out of scope (explicit)

- No change to `build.gradle.kts` except syntax cleanup if needed.
- No changes to CI / signing.
- No change to `network_security_config.xml`.
- No dependency updates.

## 6. Acceptance criteria

| ID | Test | Pass |
|----|------|------|
| AC1 | Launch, wait 10 min | No `JS watchdog TRIGGERED`, no `MAINFRAME ERROR` |
| AC2 | `adb shell am crash com.hermes.firetv:sandboxed_process0` | Dashboard back on screen within 10 s |
| AC3 | `adb reboot`; wait for launcher | DashboardActivity visible within 60 s |
| AC4 | 12 h soak on 4K Gen 1 | No reload loop; `procrank` memory ≤ 120 % of t=0 |
| AC5 | Kill network; re-enable after 2 min | Overlay shows within 20 s; dashboard back within 60 s of reconnect |
| AC6 | Force exit | Wake lock released (`dumpsys power | grep FireTVDashboard::ScreenAwake` shows no entry) |
| AC7 | Throw runtime exception in test build | `filesDir/crash-*.log` written; on next launch, POST seen in logcat; file deleted |
| AC8 | `apksigner verify -v --min-sdk-version 22` | v2 TRUE, v1 FALSE (unchanged from current) |

## 7. Risk / rollback

- All changes are local to the APK; rollback is a redeploy of the previous `app-release.apk`.
- Crash-logging POST uses a best-effort thread; network failure during POST cannot break launch.
- Renderer-crash rebuild allocates a second WebView briefly — negligible memory delta on 1 GB devices.
