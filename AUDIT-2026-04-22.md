## 1. EXECUTIVE SUMMARY (blunt)

This is **not a "fix all the bugs" situation** — it's a **deployment/release drift situation**.

The Hermes-side source tree is largely correct. The shipped artifacts are not:

1. The **live Next.js app at `/opt/data/home/assistant-stack/dashboard/`** is an older snapshot that does NOT return `unscheduled_events` from `/api/refresh`. The client code (both deployed and in-source) destructures that field and calls `.length` / `.map` on it → every Fire Stick WebView session hits a runtime TypeError after the first fetch and renders a blank page. **This is the "dashboard disappears" symptom.** Confirmed by `curl http://127.0.0.1:8081/api/refresh` (no `unscheduled_events` key) vs the hermes-side source at `assistant-stack/dashboard/src/app/api/refresh/route.ts:130` which does include it.
2. The **shipped APK `firetv-v1.8.1.apk`** is a stale build: `versionCode=1`, `versionName="1.0"`, contains `DashboardActivity$sslTrustManager$1` + `sslSocketFactory$2` classes, `android:usesCleartextTraffic="true"`, no `RECEIVE_BOOT_COMPLETED` permission, no `BootReceiver`. The **source tree has already replaced all of this** (manifest has `usesCleartextTraffic="false"`, `allowBackup="false"`, `networkSecurityConfig`, `BootReceiver` registered, `RECEIVE_BOOT_COMPLETED` permission, no TrustManager code — comment in `DashboardActivity.kt:405` explicitly says "Standard HTTPS is used throughout. No custom TrustManager"). The filename `firetv-v1.8.1.apk` and `public/version.json` saying 1.8.1/build 10 are **marketing numbers**, not what's in the APK. Because `versionCode=1` is hardcoded-via-CI-fallback, every "new" APK has been installed as the same version, so upgrade installs are silently no-ops.
3. The **release pipeline is the root cause.** `app/build.gradle.kts:31` falls back to `versionCode=1` when `CI_BUILD_NUMBER` is missing. If the APK in `public/apk/` was ever built locally (e.g. `./gradlew assembleRelease` on a laptop), it will have versionCode=1 regardless of filename. Every local build collides with every previous local build.

Once 1 + 2 + 3 are fixed (deploy the source, rebuild the APK with a real versionCode, lock the release pipeline), the app moves from "broken in production" to "kiosk-appropriate." Remaining items are incremental hardening (crash telemetry, stale-data indicator, screensaver defeat).

**You don't need to rewrite this app. You need to ship it.**

## 3. REPO FINDINGS

### 3.1 Build / packaging

**P0 — APK versionCode frozen at 1.** `aapt dump badging firetv-v1.8.1.apk` → `versionCode='1' versionName='1.0'`. The CI build number is the sole source of `versionCode` (`app/build.gradle.kts:31`), but `gradle.properties` has no `CI_BUILD_NUMBER=` line, so the fallback `?:1` wins for any local build. Filename and `public/version.json` claim 1.8.1 build 10 — cosmetic only.

**P0 — Shipped APK predates the source.** DEX contains `DashboardActivity$sslTrustManager$1` and `DashboardActivity$sslSocketFactory$2`; DashboardActivity.kt in source explicitly removed this and contains a comment (line 405) "No custom TrustManager." Shipped manifest has `usesCleartextTraffic="true"` and no `RECEIVE_BOOT_COMPLETED`; source manifest has `usesCleartextTraffic="false"` and declares the permission + BootReceiver. So the APK was built from an earlier commit and has been sitting in `public/apk/` since.

**P1 — Package size is fine.** 4.87 MB. No native libs. classes.dex 8.87 MB, classes2.dex 3 KB (mostly androidx + kotlinx-coroutines). No noise.

**P1 — minSdk 22 / target 34 / compile 34** — correct for the Fire Stick 4K Gen 1 target. Source project is already set up right.

**P1 — Next.js package.json is inconsistent.** `next: ^16.2.4, react: ^18.3.0` — Next 16 peer is React 19. It resolves today because npm's older peerDeps rules tolerate it, but a clean-slate rebuild may not. `typescript: 6.0.3` — TS 6 does not exist publicly; latest is 5.6. This pins will break any `npm ci` from cold.

**P2 — `outputFileTracingRoot: '/opt/data/home/assistant-stack/dashboard'`** hardcoded absolute path in `next.config.ts`. Non-portable but works in current deploy.

### 3.2 Fire OS compatibility

**P1 — Shipped APK has no BootReceiver.** DEX strings show no receiver class. Source manifest now declares `BootReceiver` and a `RECEIVE_BOOT_COMPLETED` intent filter — ship this.

**P1 — Foreground service type.** Source has `foregroundServiceType="specialUse"` with the `PROPERTY_SPECIAL_USE_FGS_SUBTYPE` property. Good. Verify that `FOREGROUND_SERVICE_SPECIAL_USE` permission is declared — current source manifest only declares `FOREGROUND_SERVICE`. On targetSdk 34 this is now required. *(See PRD T-APK-03.)*

**P1 — Leanback launcher intent-filter** is present in both shipped and source — good. Banner drawable exists.

**P1 — HDMI/screensaver.** Wake lock keeps CPU alive but Amazon screensaver overlays after 5 min idle. `FLAG_KEEP_SCREEN_ON` is set in source (DashboardActivity.kt:200). We still need operator instructions to disable Fire Stick screensaver, OR a periodic synthetic keypress to defeat it.

### 3.3 Runtime stability

**P0 — `/api/refresh` response shape mismatch on deployed instance.**
- Deployed: `src/app/api/refresh/route.ts` returns `{ today, today_events, week_events, upcoming_tasks }` — no unscheduled_events.
- Hermes source: `src/app/api/refresh/route.ts:130` returns `unscheduled_events: unscheduled` — plus has event archival (30-day cutoff, writes to `local_calendar_archive.json`).
- `page.tsx:499` (both versions) destructures `unscheduled_events` and passes to `TodayFocusView` / `ListView` which call `.length` (line 312), `.map` (line 315), `.slice` (line 327) on it.

When the deployed API omits the field, these throw `TypeError: Cannot read properties of undefined`. No error boundary exists, so React unmounts the entire tree to a blank DOM. `KeepAwakeService` continues to hold the wake lock, so the display stays on — showing a blank/near-blank page. This matches the user-reported "dashboard disappears."

**P0 — Task filter semantics.** Tasks JSON uses `status: "pending"|"completed"` but the deployed refresh route filters on `!t.completed` (checks a field that doesn't exist → always truthy → every task passes). Currently harmless; will silently miss the "show only incomplete tasks" filter the moment someone writes `status: "completed"`.

**P1 — No error boundary.** `src/app/layout.tsx` is 9 lines, no `<ErrorBoundary>`, no `error.tsx`, no `global-error.tsx`.

**P1 — No retry/backoff on API failures.** `fetchData` catches, sets `error=true`, returns. 30s interval keeps trying. No backoff needed — but also no visible "data is N minutes old" text. The red banner at line 570-574 says "Could not reach calendar server. Showing cached data." but Cached data may be 24h old and the user has no way to tell.

**P1 — WebView renderer recovery.** Can't confirm from DEX alone, but `onRenderProcessGone` is not in the DEX strings → not handled. If the WebView renderer crashes (low memory, aggressive pages), the whole Activity dies. `BootReceiver` only covers device boot, not Activity death.

**P2 — Client-side timezone parsing.** Multiple uses of `toLocaleString('en-US', { timeZone: 'America/New_York' })` in `page.tsx`. Modern WebView handles it; Fire OS 6 WebView (Chromium ~70) sometimes has incomplete ICU. Move timezone math server-side.

### 3.4 Display reliability

**P1 — Screensaver overlay on idle.** `FLAG_KEEP_SCREEN_ON` prevents screen-off but Amazon's screensaver is an overlay, not a sleep event. Wake lock alone doesn't defeat it. Needs periodic `dispatchKeyEvent` injection OR operator docs say "set screensaver to Never."

**P1 — No "last updated" visibility.** `page.tsx:565` renders the timestamp in `#334155` on `#0a0e1a` — invisible from 10 feet.

**P2 — First-paint is always the "Loading…" shell.** SSR sends `data: null` because fetching is client-only. Small delay on modern hardware, 2-3s on older Fire Stick. Fix: Server Component fetches JSON at request time, inlines initial data.

### 3.5 Security / config

**P0 — Custom SSL TrustManager in shipped APK.** DEX has `DashboardActivity$sslTrustManager$1` + `sslSocketFactory$2`. Combined with `usesCleartextTraffic="true"` and hardcoded `http://2.24.198.162` string, this is a textbook "trust all certs + HTTP fallback" pattern. Source tree fixed this already — rebuild will ship the fix.

**P1 — Root path not auth-gated.** Traefik `dashboard-https` router has no `middlewares` entry; only `api-https` applies `auth-check`. Family schedule is publicly readable.

**P1 — APK signing scheme.** `META-INF/CASHLAB-*.SF/.RSA` plus `CERT.SF/.RSA` — dual-signature. v1 scheme may cause `INSTALL_PARSE_FAILED_NO_CERTIFICATES` on some Fire OS 5 devices. Verify with `apksigner verify -v`.

**P1 — No crash telemetry.** Zero observability when the app fails in the field.

### 3.6 Maintainability

- **P1 — No CI evidence of successful build** in `public/apk/`. The presence of old TrustManager code suggests the APK was assembled manually outside CI.
- **P2 — Empty files** `family-dashboard@1.0.0` and `next` at the dashboard root look like stray shell output.
- **P2 — `dashboard_startup.log` shows `EADDRINUSE` on port 80** — the `npm run dev -p 80` script conflicts with Traefik. The actual running process (PID 298066) is Next standalone on :8081, not started via this script.
- **P3 — No systemd unit** for the Next.js process. It's running as root from `.next/standalone/server.js`, no auto-restart on crash.

## 4. ARCHITECTURE OPINION

Thin WebView wrapper is the right architecture for a family wall display. Keep it. The implementation already has the right bones in source (boot receiver, foreground service with wake lock, KEEP_SCREEN_ON, no TrustManager, strict network config). The gap is operational (release pipeline, deployment drift, crash telemetry, stale-data indicator), not architectural.

Alternative: **Fully Kiosk Browser** from the Amazon Appstore gives you all five kiosk essentials (boot, crash restart, renderer-gone, network recovery, update channel) for free. Recommend this ONLY if you don't want to own the APK code. Given you already have a good source tree, stick with the custom APK.
